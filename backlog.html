Agent-First Backlog Management

Markdown. Git. Worktrees. Done.

The Problem

AI agents are powerful.

But they forget everything between sessions.

Agent Memory Is a Lie

Session 1: "Build the login page"

Agent writes 400 lines. Commits. Session ends.

Session 2: "Now add password reset"

Agent has no idea what it built yesterday.

No context. No plan. No memory of decisions made.

The Coordination Problem

Agent A is refactoring the router

Agent B is adding a new route

Result: merge conflicts, duplicated work, broken builds

Shared mutable state + no coordination = chaos

The Trust Problem

  • Agents can write code… but should they merge it?
  • Agents can commit… but should they push to main?
  • Agents can be fast… but who reviews the work?


You need human gates at the right places.

The Design

A backlog system built for agents and humans.

Three Principles

  1. Everything is in git
    No database. No SaaS. No API keys. Just files.
  2. Agent-first tooling
    Stories are structured data that agents can read, write, and act on.
  3. Human gatekeeping
    Agents work in isolation. Humans review and merge.

Stories Are Markdown + Frontmatter


---
epic: E01 · Auth & Onboarding
story_id: S-42
title: Password Reset Flow
type: feature
points: 5
status: backlog
added: 2026-03-15
---

# S-42 · Password Reset Flow

## Summary
Allow users to reset their password via email...

## Acceptance Criteria
- [ ] User receives reset email within 30 seconds
- [ ] Token expires after 1 hour

Human-readable. Machine-parseable. Diff-friendly.

The Lifecycle

  ┌──────────┐      bl workon      ┌─────────────┐      bl done        ┌──────────┐
  │          │    ───────────────> │             │   ───────────────>  │          │
  │ backlog  │                     │ in_progress │                     │   done   │
  │          │  creates worktree   │             │  archives story     │          │
  └──────────┘  launches agent     └─────────────┘  syncs indexes      └──────────┘
                sets status                         sets commit hash        │
                                                                            │
                                                                       bl cleanup
                                                                            │
                                                                       merge to main
                                                                       push origin
                                                                       remove worktree

Isolation via Git Worktrees

Each agent gets its own complete copy of the repo.

The Layout

  top-secret-project/                    main repo (stays on main)
  │
  top-secret-project-worktrees/
  ├── s67-spa-navigation/                Agent 1 — own branch, own files
  ├── s66-residence-record/              Agent 2 — completely independent
  └── s65-storage-binding/               Agent 3 — no conflicts possible

Worktrees share git history but have separate working directories.

No file locking. No race conditions. No coordination overhead.

Agent Permissions

Can do:

  • Edit any file in their worktree
  • Run tests, install deps
  • Commit and push their branch
  • Read the full codebase

Cannot do:

  • Run backlog sync
  • Edit index files
  • Merge branches
  • Touch main

Agents are powerful within their sandbox. Humans control what reaches main.

In Practice

Create a Story


$ bl new --epic E01 --title "Password Reset" --points 5

New backlog story — next available ID: S-42

Created story: docs/backlog/e01-auth/S42-password-reset/index.md
Syncing backlog... done.

Or use the Claude Code skill: /bl-new --epic E01 --title "Password Reset" --points 5

Stories don't have to be pre-written. A designer can describe a feature in conversation with the agent — the agent creates the story, fills in acceptance criteria, estimates points, and scaffolds the file. The backlog becomes a collaborative artifact between humans and agents.

Start Work


$ bl workon S-42

Story    : S-42 — Password Reset
Branch   : s42-password-reset
Worktree : ../top-secret-project-worktrees/s42-password-reset

Pulling latest main...
Creating new worktree + branch 's42-password-reset' from main...
Installing dependencies in worktree...
Launching claude...

The agent gets a full prompt with the story context, knows its branch, knows what files to edit.

# Review what the agent did

Removing worktree...

Why This Works

vs. Jira / Linear / Notion

SaaS tools

  • Agents need API tokens
  • Rate limits on reads
  • Context window wasted on API calls
  • State lives outside the repo
  • Merge story? Manual status update

bl

  • Agents read files directly
  • No network required
  • Stories are in the code context
  • Single source of truth: git
  • Merge? bl cleanup handles it

vs. GitHub Issues / Projects

GitHub Issues

  • Metadata is unstructured
  • No machine-readable format
  • Disconnected from the code
  • No offline access
  • No worktree automation

bl

  • YAML frontmatter = structured data
  • Parseable by any tool
  • Lives next to the code
  • Works completely offline
  • bl workon = branch + session

vs. Custom Agent Orchestrators

Orchestrators

  • Complex infra to maintain
  • Custom state management
  • Opaque decision-making
  • Hard to debug when agents conflict

bl

  • Shell scripts + git worktrees
  • State = files on disk
  • Every decision is a commit
  • Isolation by design — no conflicts

The Key Insight

You don't need a database for agent memory.
You need structured files in a version-controlled repo
with tooling that agents already understand.

Architecture

The Complete Toolkit

Command Purpose Who
bl init Scaffold backlog in any repo Human
bl new Create a story Human / Agent
bl workon Start work in isolated worktree Human
bl list Show active worktrees Human
bl done Mark story complete Agent
bl sync Regenerate index tables Automated
bl cleanup Merge, push, clean up Human

What the Agent Sees

When bl workon launches a Claude session, it injects: 


You are working on backlog story S-42 — "Password Reset".

You are working in a git worktree at: ../worktrees/s42-password-reset
The branch is: s42-password-reset

1. Read the story file to understand the full scope
2. If not yet in_progress, update the frontmatter
3. Implement the story according to the execution plan
4. Commit with meaningful messages prefixed with "S-42: "

IMPORTANT: Do NOT run bl sync or edit backlog index files.
Sync runs on main after merge via `bl cleanup S-42`.

Full context. Clear boundaries. No ambiguity.

Everything Is Auditable


# Who changed what?
$ git log --oneline docs/backlog/

# What did the agent do on S-42?
$ git log --oneline main..s42-password-reset

# When was S-42 completed?
$ grep "completed:" docs/backlog/e01-auth/.completed/S42-*/index.md

# What's the current backlog?
$ cat docs/backlog/index.md

No dashboards needed.
git log is the audit trail.

Recap

  • Problem: Agents forget. Agents collide. Agents need guardrails.
  • Solution: Structured stories in git. Worktree isolation. Human merge gates.
  • Result: Multiple agents, parallel work, zero conflicts, full audit trail.

Get Started


# Initialize in any repo
bl init

# Create your first story
bl new --epic new --epic-title "My Epic" \
    --title "First Story" --points 3

bl workon S-1

# Review and ship
bl cleanup S-1
